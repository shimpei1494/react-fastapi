# コードレビュー観点

Linter / Formatter（Ruff・Biome）で自動検知できるものは対象外。

---

## 共通

### 可読性
- 関数・変数名が処理内容を適切に表しているか（略語の乱用、汎用すぎる名前 `data`, `result`, `tmp` の回避）
- 1つの関数が1つの責務に収まっているか（肥大化した関数への分割検討）
- 複雑な条件式やネストが深い処理に説明コメントがあるか

### 命名
- 関数名が動詞始まりか（`get_`, `create_`, `update_`, `delete_` / `use`, `handle`, `fetch` など）
- bool を返す関数・変数が `is_`, `has_`, `can_` などで識別できるか
- 定数はすべて大文字スネークケースか（Python: `MAX_RETRY`, TS: `MAX_RETRY`）

---

## バックエンド（FastAPI / Python）

### セキュリティ
- 外部入力（クエリパラメータ・リクエストボディ）を直接 SQL に渡していないか（SQLAlchemy ORM を使っていれば基本 OK）
- `.env` や設定値をハードコードしていないか（`pydantic-settings` 経由で読み込むこと）
- エラーレスポンスにスタックトレースや内部情報を含めていないか

### パフォーマンス
- N+1 クエリが発生していないか（ループ内で DB アクセスしていたら `joinedload` / `selectinload` を検討）
- 不要なフルテーブルスキャンが起きていないか（検索・フィルタに使うカラムにインデックスがあるか）

### API 設計
- HTTPステータスコードが適切か（作成は `201`、存在しないリソースは `404`、バリデーションエラーは `422` など）
- Pydantic スキーマで入力バリデーションを行っているか（生の dict を受け取っていないか）
- ページネーションが必要なリストエンドポイントで全件返していないか

---

## フロントエンド（React / TypeScript）

### セキュリティ
- `dangerouslySetInnerHTML` を使っている場合、入力をサニタイズしているか
- API キーや機密情報をクライアントコードに含めていないか

### パフォーマンス
- 重い計算処理を `useMemo`、コールバックを `useCallback` でメモ化しているか（ただし過剰なメモ化も避ける）
- リスト描画で `key` に配列インデックスを使っていないか（ユニークな ID を使う）
- 不要な再レンダリングが発生していないか（親の state 変更が無関係な子に伝播していないか）

### React / Hooks
- `useEffect` を使っていないか（イベントハンドラ・派生 state・カスタムフック内のデータフェッチなどで代替できないか検討する）
  - 外部システムとの同期（WebSocket・サードパーティライブラリの初期化など）は許容
  - データフェッチは `useEffect` ではなくカスタムフックやライブラリ（SWR・TanStack Query など）で行う
- カスタムフックがコンポーネントと UI に関係しないロジックを適切に分離しているか

### 型安全
- API レスポンスの型が定義されているか（`unknown` で受け取って型ガードするか、スキーマから生成した型を使う）

---

## テスト

- ハッピーパスだけでなく、異常系（空値・境界値・エラーレスポンス）のケースがあるか
- テストがテスト対象の実装詳細に依存しすぎていないか（内部メソッドを直接テストするより振る舞いをテストする）
- モックが現実と乖離していないか（実際のAPIレスポンス形式と合っているか）
